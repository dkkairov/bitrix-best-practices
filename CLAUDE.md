# CLAUDE.md — Схема базы знаний «Лучшие практики внедрения Bitrix24»

Это **схема** (config-слой) LLM Wiki по методу Андрея Карпати
([llm-wiki.md](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)).
Claude Code автозагружает этот файл, поэтому правила ниже действуют **всегда** — при любой
операции с вики (ingest / query / lint и ручном редактировании).

> Если правила в этом файле противоречат разовому указанию пользователя — приоритет у указания
> пользователя. Об отклонении от схемы коротко сообщай.

---

## 1. Назначение и аудитория

- **Что:** накапливающаяся база знаний по лучшим практикам **внедрения Bitrix24** (облако и коробка).
- **Для кого:** команда / агентство внедрения. Тон — рабочий, прикладной. Цель — стандартизация:
  единые чек-листы, паттерны, антипаттерны, рецепты и регламенты, на которые опираются разные
  специалисты.
- **Принцип Карпати:** вики — это **persistent, compounding artifact**. Человек приносит источники
  и задаёт вопросы; LLM ведёт рутину — конспектирует, перелинковывает, ищет противоречия и
  устаревание. Не RAG «с нуля на каждый запрос», а растущий, поддерживаемый артефакт.
- **Язык контента:** русский. **Идентификаторы/слаги/пути:** латиница `kebab-case`.

---

## 2. Три слоя (+ служебные файлы)

1. **`raw/`** — неизменяемые источники (статьи, выгрузки доков, скриншоты, ретро проектов). Не правим.
2. **`wiki/`** — сгенерированные страницы: конспекты источников, сущности, концепты, чек-листы,
   паттерны, рецепты, аналитика.
3. **`CLAUDE.md`** — эта схема.

Служебные: **`index.md`** (каталог-навигатор), **`log.md`** (журнал операций), `README.md` (для людей),
git (история версий).

## 3. Карта каталогов

```
raw/sources/            источники (по файлу/папке на источник, immutable)
raw/assets/             медиа из источников
wiki/overview.md        вводная по всей вики
wiki/_templates/        шаблоны страниц по типам (копировать при создании)
wiki/modules/<m>/       ПРОДУКТОВЫЕ модули (облако и коробка): crm, tasks-projects, bizproc,
                        smart-process, telephony, sites-stores, communications, collaboration,
                        ai-copilot, hr, rest-integrations, marketplace-apps, permissions, administration
                          └ внутри — подпапки решений: guides/ patterns/ antipatterns/ reference/ (§4.5)
wiki/development/<a>/   КОРОБКА (разработка): core-d7, components, modules-custom,
                        templates-design, performance, migrations, server-admin
                          └ внутри — те же подпапки решений (§4.5)
wiki/cross-cutting/     playbooks (жизненный цикл), patterns, antipatterns
wiki/glossary/          сущности и термины (entity-страницы)
wiki/sources/           конспекты источников (source-summary)
```

В каждой папке-модуле/области — `_index.md` (хаб): назначение, список страниц, статус покрытия.

## 4. Соглашения страниц

### 4.1. Именование файлов
- Формат имени контент-страницы: **`<id>-<слаг>.md`**, где `<id>` — номер решения в нижнем регистре
  (см. §4.5), `<слаг>` — латиница `kebab-case`. Пример:
  `m01-pt01-crm-sales-funnel-design-pattern.md`. Имя файла (с префиксом ID) **глобально уникально** —
  Obsidian резолвит `[[ссылки]]` по имени файла без пути.
- Хабы папок — всегда `_index.md` (без ID). Без ID также: шаблоны `_templates/`, `wiki/overview.md`,
  `index.md`, служебный `wiki/sources/sources-backlog.md`.

### 4.2. Frontmatter (обязателен на каждой странице)
```yaml
---
id: "M01-CL01"                           # номер решения (см. §4.5); у хабов/шаблонов — отсутствует
title: "Чек-лист запуска CRM"            # человекочитаемый RU-заголовок
type: checklist                          # см. 4.3
module: crm                              # slug папки-модуля/области (семантический тег, ср. §4.5)
edition: both                            # cloud | box | both
status: verified                         # draft | verified | deprecated
confidence: high                         # high | medium | low
provenance: empirical                    # documented | empirical | mixed
verified: "2026-06-19 / Bitrix24 cloud"  # когда/где проверено (обязательно при status: verified)
tags: [воронка, запуск, онбординг]
sources: ["[[s-ss01-src-b24-crm-deal-add]]"]   # конспекты из wiki/sources (если есть)
related: ["[[m01-pt01-crm-sales-funnel-design-pattern]]", "[[x03-ap01-crm-stage-explosion]]"]
aliases: ["crm-launch-checklist"]        # прежний слаг(и) — обратная совместимость [[ссылок]]
updated: "2026-06-19"
---
```

### 4.3. Типы страниц (`type`) и когда какой
| type | Назначение | Папка решения | Префикс |
|------|-----------|----------------|---------|
| `checklist` | Пошаговый проверяемый список (запуск, аудит, приёмка) | `guides/` | `CL` |
| `recipe` | Конкретное «как сделать» (настройка/код шаг-за-шагом) | `guides/` | `RC` |
| `pattern` | Проверенное решение задачи + контекст применения | `patterns/` | `PT` |
| `antipattern` | Типичная ошибка, чем вредит, чем заменить | `antipatterns/` | `AP` |
| `concept` | Обзорная/тематическая страница | `reference/` | `CN` |
| `entity` | Конкретный объект: REST-метод, событие, робот, приложение, термин | `reference/` (в glossary — плоско) | `EN` |
| `source-summary` | Конспект одного источника | `wiki/sources/` (плоско) | `SS` |

Папки `guides/patterns/antipatterns/reference/` — внутри `wiki/modules/<m>/` и `wiki/development/<a>/`
(см. §4.5). В `cross-cutting/`, `glossary/`, `sources/` — особое размещение (тоже §4.5).

### 4.4. Стиль
- Кратко и по делу. Утверждения — **проверяемые**; примеры конфигов/кода — в блоках.
- Каждая практика отвечает на: **в чём суть → когда применять → как сделать → чего избегать**.
- Указывай **edition** прямо в тексте, если поведение облака и коробки расходится.
- Никаких «фактов по памяти» про REST/события без сверки — см. раздел 9.

### 4.5. Нумерация и адресация (ID)

Каждая контент-страница получает **стабильный ID**, встроенный в имя файла и слаг (см. §4.1).
Формат: **`<РАЗДЕЛ>-<ТИП><NN>`** → `M01-PT01`, `D01-CN01`, `X03-AP02`, `G-EN01`, `S-SS01`.

**Коды разделов** (присваиваются по фактической папке файла — дерево = источник истины):

| Префикс | Раздел | Папки на диске |
|---------|--------|----------------|
| `M01`–`M14` | Продуктовые модули | `wiki/modules/<m>/` |
| `D01`–`D07` | Разработка коробки | `wiki/development/<a>/` |
| `X01`–`X03` | Сквозное | `wiki/cross-cutting/{playbooks,patterns,antipatterns}/` |
| `G` | Глоссарий (один раздел) | `wiki/glossary/` |
| `S` | Источники (один раздел) | `wiki/sources/` |

Привязка кодов (фиксированный реестр, порядок — как в §3):
`crm=M01, tasks-projects=M02, bizproc=M03, smart-process=M04, telephony=M05, sites-stores=M06,
communications=M07, collaboration=M08, ai-copilot=M09, hr=M10, rest-integrations=M11,
marketplace-apps=M12, permissions=M13, administration=M14`;
`core-d7=D01, components=D02, modules-custom=D03, templates-design=D04, performance=D05,
migrations=D06, server-admin=D07`; `playbooks=X01, patterns=X02, antipatterns=X03`.

**Под-префиксы типов:** `CL` checklist · `RC` recipe · `PT` pattern · `AP` antipattern ·
`CN` concept · `EN` entity · `SS` source-summary.

**Счётчик `NN`** — двузначный, в пределах пары (раздел × тип), монотонный; **номера не
переиспользуются** (deprecated-страница сохраняет номер). Новый номер = (max существующий для пары) + 1.

**Папки решений внутри `modules/<m>/` и `development/<a>/`:**

| Папка | Типы (под-префиксы) |
|-------|---------------------|
| `guides/` | checklist `CL`, recipe `RC` |
| `patterns/` | pattern `PT` |
| `antipatterns/` | antipattern `AP` |
| `reference/` | concept `CN`, entity `EN` |

- Подпапку создаём **по факту первой страницы** её группы — пустых не плодим.
- `cross-cutting/` уже разделён по назначению (`playbooks/`, `patterns/`, `antipatterns/`) — внутри плоско.
- `glossary/` и `sources/` — плоско (один тип на раздел).
- Поле `module` во frontmatter — семантический тег, **может не совпадать** с префиксом раздела
  (например, CRM-антипаттерн в `cross-cutting/antipatterns/` → `X03-AP0N`, но `module: crm`).
- **`aliases`:** при переименовании/смене ID сохраняй прежний слаг в `aliases` — старые `[[ссылки]]`
  продолжат резолвиться в Obsidian.

## 5. Перелинковка

- Связи **между контент-страницами** — Obsidian-wikilinks по имени файла `[[<id>-слаг]]`. В теле
  используй подпись для читаемости:
  `[[m01-pt01-crm-sales-funnel-design-pattern|Проектирование воронки]]`; в `related` достаточно
  голого ключа `[[m01-pt01-crm-sales-funnel-design-pattern]]`.
- **`aliases`** в frontmatter хранят прежние слаги — старые ссылки не ломаются при переименовании
  (см. §4.5).
- Ссылки **на хабы** `_index.md` — обычными относительными markdown-ссылками
  (`[CRM](_index.md)`, `[CRM](modules/crm/_index.md)`): имя `_index` неуникально и для `[[...]]` не
  годится.
- **Правило связности:** у каждой контент-страницы ≥ 1 входящая ссылка (минимум из `_index.md` своей папки).
  Страница без входящих ссылок — **орфан** (чинит lint).
- Хабы `_index.md` перечисляют все страницы папки. `index.md` — глобальные представления.
- Двусторонность: добавил `related` → проверь, что обратная ссылка уместна на связанной странице.
- Упомянул сущность/концепт, у которого есть (или должна быть) страница в `glossary/` — ставь ссылку.

## 6. Доверие и жизненный цикл (важно для Bitrix24)

Bitrix24 — быстро меняющийся продукт; у практик есть «срок годности».

- **`status`:** `draft` (черновик) → `verified` (проверено) → `deprecated` (устарело, но оставляем
  с пометкой и ссылкой на замену).
- **`verified`:** дата + где проверено (`cloud` / версия коробки). Обязательно при `status: verified`.
- **`provenance`:** `documented` (есть в офф. доке, со ссылкой) / `empirical` (наш опыт, воркэраунд) /
  `mixed`. Эмпирические утверждения помечай явно.
- **`confidence`:** субъективная уверенность (`high/medium/low`).
- **Противоречия:** если новый источник противоречит странице — не удалять молча. Обновить с пометкой
  что изменилось и когда, при необходимости старое — в `deprecated`. Зафиксировать в `log.md`.
- **Облако ≠ коробка:** не переноси практику между edition без проверки. Помечай `edition`.

## 7. Форматы вывода (для query)

По умолчанию — markdown. По запросу: таблицы сравнения, чек-листы, **клиентские регламенты/методички**
(тон для заказчика, без внутреннего жаргона), презентации (Marp), схемы. Всегда — со ссылками на
страницы-источники вики.

## 8. Рабочие процессы

Операции реализованы как команды Claude Code в `.claude/commands/wiki/` — `/wiki:ingest`,
`/wiki:query`, `/wiki:lint`. Краткая суть (детали — в файлах команд):

- **ingest `<источник>`** — положить в `raw/sources/`, прочитать целиком, обсудить выводы, создать
  `source-summary`, обновить `index.md`, найти и обновить 10–15 связанных страниц, пометить
  противоречия, дописать `log.md`. REST/события — сверить через MCP (раздел 9).
- **query `<вопрос>`** — прочитать `index.md`, найти/прочитать релевантные страницы, синтезировать
  ответ со ссылками в нужном формате; ценные находки — дофайлить новыми страницами; записать в `log.md`.
- **lint** — health-check: противоречия, устаревшее (старый `verified`), ошибки `edition`, орфаны,
  упомянутые-но-не-созданные концепты, недостающие перелинковки, REST без сверки, нумерация/ID
  (формат, уникальность, имя↔id, префикс↔папка), пробелы покрытия.

**После любой правки вики:** обнови `updated`, при необходимости `index.md`, добавь запись в `log.md`.

## 9. Доменные правила Bitrix24

- **Сверка REST/событий — обязательна.** Для методов и событий используй MCP-сервер документации
  Bitrix24 (`bitrix-search`, `bitrix-method-details`, `bitrix-event-details`,
  `bitrix-article-details`) и Context7. Указывай актуальный источник — домен **`apidocs.bitrix24.ru`**.
  Не выдумывай имена методов/полей.
- **Скоупы и права:** для REST-практик указывай требуемый `scope` и право доступа.
- **Облако vs коробка:** разработка на PHP/D7, агенты, cron, правка ядра — это **коробка**
  (`wiki/development/`). Облако — no-code + REST/вебхуки + приложения маркетплейса.
- **Не править ядро коробки** — паттерн событий/`init.php`/локальных модулей. Это сквозной принцип.
- **Цитируй версии:** где практика зависит от версии модуля/редакции — указывай явно в `verified`.

## 10. Быстрый старт по операциям

```
/wiki:ingest <url|путь|вставленный текст>   # добавить и встроить источник
/wiki:query  <вопрос>                       # ответить по вики со ссылками
/wiki:lint                                  # проверить здоровье вики
```

Точка входа для навигации — [[overview]] и `index.md`.
