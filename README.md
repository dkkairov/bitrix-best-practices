# База знаний: Лучшие практики внедрения Bitrix24

Накапливающаяся база знаний (БЗ) по внедрению **Bitrix24** (облако и коробка), построенная по
методу **LLM Wiki** Андрея Карпати
([gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)).

Идея метода: вики — это не разовый ответ, а **растущий артефакт**. Вы приносите источники и задаёте
вопросы, а ассистент (Claude Code) ведёт рутину: конспектирует, перелинковывает страницы, следит за
противоречиями и устареванием. Со временем БЗ становится тем ценнее, чем больше в неё вложено.

## Как устроено (три слоя)

| Слой | Папка | Что внутри |
|------|-------|-----------|
| Источники | `raw/` | Неизменяемые исходники: статьи, выгрузки доков, скриншоты, ретро проектов |
| Вики | `wiki/` | Сгенерированные страницы: чек-листы, паттерны, антипаттерны, рецепты, конспекты, глоссарий |
| Схема | `CLAUDE.md` | Правила: соглашения, frontmatter, перелинковка, жизненный цикл, рабочие процессы |

Служебное: `index.md` — каталог-навигатор, `log.md` — журнал операций.

## Навигация

- Начните с [обзора вики](wiki/overview.md) и [каталога `index.md`](index.md).
- В каждой папке-модуле есть хаб `_index.md` со списком страниц и статусом покрытия.
- Удобнее всего просматривать в **Obsidian** (папка уже открыта как vault): связи `[[...]]` кликабельны,
  работает граф.

## Как пользоваться (операции метода)

В Claude Code доступны три команды (см. `.claude/commands/wiki/`):

| Команда | Что делает |
|---------|-----------|
| `/wiki:ingest <url \| путь \| текст>` | Добавляет источник в `raw/`, делает конспект и встраивает знания в вики |
| `/wiki:query <вопрос>` | Отвечает по вики со ссылками; ценные находки дофайливает страницами |
| `/wiki:lint` | Health-check: противоречия, устаревшее, орфаны, пробелы покрытия |

Можно и просто попросить словами — схема `CLAUDE.md` загружается автоматически.

## Соглашения (кратко; полностью — в `CLAUDE.md`)

- **Контент — на русском**, заголовок в `title:`.
- **Нумерация решений:** у каждой страницы стабильный ID `<РАЗДЕЛ>-<ТИП><NN>` (напр. `M01-PT01`),
  встроенный в имя файла и слаг: `m01-pt01-crm-sales-funnel-design-pattern.md` (латиница `kebab-case`).
  Коды разделов (`M01–M14`, `D01–D07`, `X01–X03`, `G`, `S`) и под-префиксы типов — в `CLAUDE.md` §4.5.
- **Структура решений:** внутри модулей/областей — подпапки `guides/` `patterns/` `antipatterns/`
  `reference/` (по типу решения).
- На каждой странице — **frontmatter** с `id`, `type`, `edition` (`cloud`/`box`/`both`), `status`,
  `verified` (когда проверено), `aliases` (прежние слаги).
- Связи — wikilinks `[[<id>-слаг]]`; в `index.md` есть представление «По ID» и реестр разделов.
  У каждой страницы должна быть хотя бы одна входящая ссылка.
- **Облако ≠ коробка**: практики помечаются по применимости; разработка (PHP/D7) живёт в
  `wiki/development/`.
- REST-методы и события **сверяются с официальной документацией** (`apidocs.bitrix24.ru`) через MCP,
  а не пишутся по памяти.

## Для команды

- Перед добавлением проверьте, нет ли уже страницы на эту тему (`/wiki:lint` показывает дубликаты и
  пробелы).
- Спорные/непроверенные утверждения помечайте `status: draft` и `provenance: empirical`.
- Любая правка → обновите `updated`, при необходимости `index.md`, добавьте строку в `log.md`.
