# Миграции как код

**Результат:** изменения структуры и данных (инфоблоки, HL-блоки, пользовательские поля, опции,
агенты) воспроизводимо применяются на dev → staging → коробках клиентов.

## Зачем
Кликать одни и те же настройки руками на каждом портале — источник ошибок и рассинхрона. Миграция
описывает изменение **кодом** и применяется одинаково везде, рядом с обновлением модуля
([[recipe-module-versioning-and-private-distribution|дистрибуция]]).

## Инструмент
De-facto стандарт — модуль **sprint.migration** (ставится как обычный модуль). Каждая миграция —
класс с методами `up()` (применить) и `down()` (откатить).

```php
public function up(): void
{
    // создать UF-поле / HL-блок / задать опцию и т.п. через API D7
}
public function down(): void
{
    // обратное действие
}
```

## Что выносить в миграции
- Структуру: инфоблоки/HL-блоки, пользовательские поля (UF), типы, свойства.
- Значимые опции и справочные данные.
- Изменения, которые **эволюционируют** между версиями (в отличие от разовой регистрации модуля в `install`).

## Рабочий процесс
1. Разработчик создаёт миграцию, коммитит в git вместе с кодом модуля.
2. На каждой среде после выкладки кода — применить новые миграции (`up`).
3. Откат — `down` соответствующей миграции.

## Проверка результата
- На чистой среде серия миграций воспроизводит структуру с нуля; на существующей — доводит до
  актуальной версии без дублей.

## Откат и проблемы
- Необратимые изменения данных → делать резервную копию перед `up`, предусматривать `down`.
- «У клиента структура отличается» → не править руками, а догнать миграциями; вести реестр версий.
