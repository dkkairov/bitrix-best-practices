# Версии модуля и приватная дистрибуция

**Результат:** один и тот же модуль быстро и предсказуемо ставится и обновляется на множестве
клиентских коробок.

## Предусловия
- Готовый устанавливаемый модуль — [[d03-rc01-module-structure-and-install|Структура и установка]].

## Версионирование
- Семантические версии в `install/version.php` (`MAJOR.MINOR.PATCH`); дата — `VERSION_DATE`.
- **Код** обновляется заменой файлов модуля; **структура БД/данные** — отдельными миграциями
  ([[d06-rc01-migrations-as-code|Миграции как код]]), а не правкой `install` задним числом.
- Ведём CHANGELOG; в git — теги версий.

## Приватная дистрибуция (свои клиенты)
Канал — **ваш**, без публичного маркетплейса:
1. **Git-репозиторий модуля** (отдельный на модуль). Тег = версия.
2. Поставка на коробку клиента одним из способов:
   - `git pull` каталога `local/modules/vendor.module` на сервере клиента;
   - выкладка собранного архива (CI собирает артефакт по тегу);
   - `composer` с приватным репозиторием (опционально, через installers для bitrix-модулей).
3. Установка/обновление: положить код в `local/modules/`, установить через
   *Marketplace → Установить* или скриптом; затем применить миграции.
4. Обновление у всех клиентов = новый тег → выкатка по вашему каналу.

> Массовое **авто**-обновление «у всех сразу» из коробочного модуля штатно даёт лишь
> marketplace.1c-bitrix.ru (публикация+модерация). Если нужно именно это или клиенты на **облаке** —
> рассматривайте тиражное REST-приложение (другой паттерн).

## Проверка результата
- На чистой коробке: установка с нуля по тегу проходит; на существующей: обновление + миграции без потерь.
- Версия в админке соответствует тегу git.

## Откат и проблемы
- Рассинхрон версий у клиентов → веди реестр «клиент → версия», автоматизируй выкатку
  ([[d07-cl01-dev-environment-and-git|Окружение и git]]).
- Конфликты при `git pull` на проде → деплой только собранными артефактами, не правкой на месте.
