# Библиотека модулей агентства: монорепо, поставка набором, отключение

## Проблема и контекст
Агентство внедряет коробку в **одной отрасли**, поэтому у большинства клиентов **набор модулей
повторяется**. Держать каждый модуль в отдельном репозитории и собирать набор вручную — накладно:
кросс-модульные правки рассинхронизируются, общий код дублируется, теряется учёт «что у кого стоит».

## Решение
Вести **единый репозиторий-библиотеку** всех отраслевых модулей как **продукт агентства**:
- **Монорепо** `agency-bitrix-modules/` со всеми модулями в раскладке, совместимой с `local/modules/`.
- **Базовый модуль** `agency.core` — общий код, утилиты, общий Composer-`vendor/`; остальные модули
  зависят от него (`Loader::includeModule('agency.core')`).
- **Поставка набором:** на коробку клиента кладётся вся библиотека, **устанавливаются только нужные**
  модули, лишние — не ставятся (или деинсталлируются).
- **Единая версия библиотеки** — одна семантическая версия на весь набор (релиз «пакета»),
  не на каждый модуль.

```text
agency-bitrix-modules/        (git: единый репозиторий = продукт)
├── agency.core/              базовый модуль: общий код + composer vendor/
├── agency.crmflow/           отраслевой модуль (зависит от agency.core)
├── agency.docs/              отраслевой модуль
├── agency.reports/           …
├── VERSION                   единая версия библиотеки (semver)
├── CHANGELOG.md
└── build/                    CI: сборка артефакта релиза
```

## Когда применять
- Узкая специализация (одна отрасль) → набор модулей у клиентов **повторяется**.
- Модули **связаны** (общий код, кросс-зависимости), эволюционируют вместе.
- Нужен предсказуемый учёт «клиент → версия библиотеки».

## Когда НЕ применять
- Модули **независимы**, релизятся врозь, у разных отраслей/заказчиков → отдельные репозитории.
- Нужно дать подрядчику доступ к одному модулю, не открывая остальные → вынеси его в свой репозиторий.

## Как реализовать
1. **Монорепо + базовый модуль.** `agency.core` держит общий код и общий Composer-`vendor/`
   (см. [[recipe-composer-third-party-libraries|Composer-пакеты]]); отраслевые модули объявляют
   зависимость от него.
2. **Зависимости** проверяй в `DoInstall` (`Loader::includeModule('agency.core')` → если нет, не давать
   установку зависящего модуля).
3. **Единая версия.** Тег git = версия библиотеки; в каждом `install/version.php` — та же версия
   набора; ведёшь `CHANGELOG`.
4. **Поставка** по приватному каналу — git-тег / собранный артефакт
   (см. [[recipe-module-versioning-and-private-distribution|приватная дистрибуция]]).
5. **Установка у клиента:** код всей библиотеки в `local/modules/`, через *Marketplace → Установить*
   активируешь **только нужные** модули.
6. **Реестр** «клиент → версия библиотеки → установленные модули» — чтобы управлять обновлениями.

## Отключение модулей
Наличие файлов в `local/modules/` **≠** активный модуль. Активностью управляешь установкой/деинсталляцией:
- **Не нужен у клиента** → не устанавливай (файлы лежат, `Loader::includeModule` вернёт `false`).
- **Временно выключить, сохранив данные** → флаг-опция внутри модуля, а не `DoUninstall`
  (он предложит удалить таблицы).
- **Совсем убрать** → `DoUninstall` (снять обработчики, при необходимости — данные).
- ⚠️ Соблюдай зависимости: не отключай `agency.core`, пока стоят зависящие модули. Обработчики
  событий снимай при деинсталляции, иначе «мёртвый» код продолжит срабатывать.

## Последствия
- **Плюсы:** один источник правды, атомарные кросс-модульные изменения, переиспользование общего
  кода, простой учёт версий, быстрый онбординг нового клиента «набором».
- **Минусы/компромиссы:** отраслевые модули **зависят от `agency.core` → не автономны по отдельности**
  (сознательный размен против «изоляции/раздельной дистрибуции» из
  [[pattern-module-based-development-standard]]; «изоляция» там = от ядра, межмодульные зависимости
  объявляются и проверяются в `DoInstall`); весь набор виден внутри репо (нет изоляции доступа к одному
  модулю); единая версия → точечное обновление одного модуля = релиз всей библиотеки; на коробке клиента
  лежат и неустановленные модули (немного «лишних» файлов).

## Альтернативы
- **Multi-repo** (репо на модуль) — гибче для независимых модулей, но дороже в поддержке набора.
- **Гибрид** — монорепо для ядра и общих модулей + отдельные репо для редких/клиент-специфичных.
