# Версии модуля и приватная дистрибуция

**Результат:** библиотека модулей (единый набор, единая версия) быстро и предсказуемо ставится и
обновляется на множестве клиентских коробок.

## Предусловия
- Готовый устанавливаемый модуль — [[recipe-module-structure-and-install|Структура и установка]].

## Версионирование
- **Единая семантическая версия библиотеки** (`MAJOR.MINOR.PATCH`) на весь набор модулей; тег git =
  версия библиотеки. `install/version.php` каждого модуля несёт версию набора; дата — `VERSION_DATE`.
- **Код** обновляется заменой файлов; **структура БД/данные** — отдельными миграциями
  ([[recipe-migrations-as-code|Миграции как код]]), а не правкой `install` задним числом.
- Ведём один `CHANGELOG` на библиотеку; в git — теги версий.

## Приватная дистрибуция (свои клиенты)
Канал — **ваш**, без публичного маркетплейса. По умолчанию — **единый репозиторий-библиотека** модулей
(монорепо, см. [[pattern-module-library-monorepo|Библиотека модулей агентства]]), а не отдельный репо
на каждый модуль:
1. **Единый git-репозиторий-библиотека** всех модулей. Тег = версия библиотеки.
2. Поставка на коробку клиента одним из способов:
   - `git pull` каталога `local/modules/` (вся библиотека) на сервере клиента;
   - выкладка собранного артефакта по тегу (CI);
   - `composer` с приватным репозиторием (опционально, через installers для bitrix-модулей).
3. Установка: положить код библиотеки в `local/modules/`, **установить только нужные** модули через
   *Marketplace → Установить* или скриптом; лишние не ставить/деинсталлировать; затем применить миграции.
4. Обновление у всех клиентов = новый тег библиотеки → выкатка по вашему каналу.

> **Исключение — отдельный репозиторий на модуль** оправдан для автономных/разнородных модулей
> (другая отрасль, независимые релизы, передача подрядчику). Тогда версия ведётся на модуль.

> Массовое **авто**-обновление «у всех сразу» из коробочного модуля штатно даёт лишь
> marketplace.1c-bitrix.ru (публикация+модерация). Если нужно именно это или клиенты на **облаке** —
> рассматривайте тиражное REST-приложение (другой паттерн).

## Проверка результата
- На чистой коробке: установка с нуля по тегу проходит; на существующей: обновление + миграции без потерь.
- Версия в админке соответствует тегу git.

## Откат и проблемы
- Рассинхрон версий у клиентов → веди реестр «клиент → версия библиотеки + установленные модули»,
  автоматизируй выкатку ([[checklist-dev-environment-and-git|Окружение и git]]).
- Конфликты при `git pull` на проде → деплой только собранными артефактами, не правкой на месте.
