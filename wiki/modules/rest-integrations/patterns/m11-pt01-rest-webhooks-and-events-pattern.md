# Паттерн: вебхуки и события — как строить интеграции

## Проблема и контекст
Нужно связать Bitrix24 с внешней системой: либо внешняя система меняет данные в B24, либо B24
уведомляет внешнюю систему об изменениях. Важно выбрать правильный механизм и не потерять данные.

## Три механизма
| Механизм | Направление | Когда |
|----------|-------------|-------|
| **Входящий вебхук** | внешняя система → REST B24 | простые серверные интеграции «для себя», без OAuth; права = у создателя вебхука |
| **Исходящий вебхук** | B24 → ваш URL по событию | реакция на изменения (например, `ONCRMDEALUPDATE`) без приложения |
| **Приложение + `event.bind`** | подписка из приложения | тиражные/сложные интеграции, OAuth-права — см. [[m11-rc01-rest-oauth-app-setup|OAuth-приложение и токены]] |

## Решение (рекомендации)
- **Выбор механизма:** внутренняя интеграция «для себя» → входящий вебхук. Реакция на события без
  приложения → исходящий вебхук. Тиражное приложение → `event.bind` + OAuth.
- **События отдают минимум данных** (как правило — идентификаторы и тип события). Получив событие,
  **дозапрашивай полные данные методом REST** (например, после `ONCRMDEALUPDATE` → `crm.deal.get`).
- **Идемпотентность и ретраи:** внешняя система может получить событие повторно — обрабатывай
  безопасно к дублям; на свою сторону ставь приём в очередь и подтверждение.
- **Безопасность:** для событий приложения проверяй `application_token` (выдаётся при установке,
  событие `OnAppInstall`). Вебхук-код — секрет, не светить в клиенте.
- **Лимиты:** учитывай ограничения частоты REST; тяжёлые выгрузки — батчами.

## Как реализовать (реакция на изменение сделки)
1. Создай исходящий вебхук (или подпишись через `event.bind`) на нужное событие
   (обзор — «События при работе со сделками», `apidocs.bitrix24.ru`).
2. Прими POST на свой URL, возьми из него ID и тип события.
3. Дозапроси полные данные методом REST (`crm.deal.get`).
4. Обработай идемпотентно, подтверди приём.

## Когда НЕ применять
- Для логики **внутри** B24 (задачи, уведомления, движение по стадиям) — это роботы/триггеры
  ([[g-en02-robots-triggers|Роботы и триггеры]]), а не внешняя интеграция.

## Последствия
- **Плюсы:** надёжная, расширяемая интеграция без правки ядра.
- **Минусы:** нужно самому обеспечить надёжную доставку и идемпотентность на своей стороне.
