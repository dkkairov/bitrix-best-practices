# Рецепт: OAuth-приложение и работа с токенами

**Результат:** приложение, авторизующееся в Bitrix24 по OAuth 2.0, с корректным хранением и
автопродлением токенов.

## Когда это нужно
- Тиражные приложения и интеграции, которым нужны OAuth-права (а не один входящий вебхук).
- Для простых внутренних интеграций «для себя» достаточно вебхука — см.
  [[rest-webhooks-and-events-pattern]].

## Предусловия
- Доступ к разделу разработчика (создание приложения).
- Понимание нужных **scope** (список scope — в разделе «Доступные скоупы» на `apidocs.bitrix24.ru`).

## Шаги
1. **Создай приложение** (локальное или тиражное), укажи нужные `scope` и адрес обработчика.
2. Получи `client_id` и `client_secret`.
3. Реализуй **OAuth-обмен**: по коду авторизации получи `access_token` и `refresh_token`
   (см. «Получение/упрощённый вариант получения токенов OAuth 2.0»).
4. **Храни токены** надёжно (не в клиенте). `access_token` живёт ограниченно (порядка часа).
5. **Автопродление:** при истечении обновляй `access_token` по `refresh_token`
   (см. «Автоматическое продление токенов OAuth 2.0»).
6. При установке приложения сохрани `application_token` из события `OnAppInstall` — он нужен для
   проверки подлинности входящих событий приложения.

> Для frontend-приложений во фрейме B24 текущие данные авторизации доступны через `BX24.getAuth()`,
> принудительное обновление — `BX24.refreshAuth()`.

## Проверка результата
- Тестовый вызов метода (например, `crm.deal.add` — см. [[src-b24-crm-deal-add]]) проходит с
  актуальным токеном.
- После истечения `access_token` интеграция сама продлевает доступ без ручного вмешательства.

## Откат и проблемы
- `expired_token` / `401` → не обновляется токен: проверь логику refresh и срок.
- `insufficient_scope` / `403` → не хватает scope: добавь нужный scope в приложении.
- `QUERY_LIMIT_EXCEEDED` → превышены лимиты частоты: добавь паузы/батчи.
